Netty in action

目前对Netty是一点不懂，但是作为NIO的框架还是很有必要学的，那么我们开始吧。

再开始之前

我们先说说常见的概念，同步/异步，阻塞/非阻塞。

弄明白这几个概念比较重要，同步和异步是针对应用程序和内核而言的。
同步指的是当用户触发IO操作后等待或者轮询操作是否完成，由用户自发的去查看。
异步指的是当用户请求之后就开始忙别的事情，当任务处理完毕后内核会给用户反馈结果。

阻塞和非阻塞是针对用户访问数据而言的，说白了是根据状态不同来进行不同的操作。阻塞状态下读取或者写入会一直等待，而非阻塞方式下，读取或者写入会立即返回一个状态值。

Channel

Channel是Netty最核心的接口，一个Channel就是一个连接Socket的通道，通过Channel，你可以对Socket做各种操作。

一个网络的连接，或者是一个能够进行操作I/0操作的组件。比如说，读、写、连接、绑定。

通道提供的功能：

  • 当前通道的状态(是否为打开？是否被连接？)
  • channel的配置参数（接收buffer的大小）
  • channel支持的I/O操作
  • 和channel一起处理所有IO事件和请求的channelPipeline.(使用channelHandler来处理channel,使用channelPipeline来处理channelHandler)

Netty中所有IO的操作都是异步的，这就意味着任何IO请求都会立即返回，但是并不保证在请求结束时，请求的操作都会结束。但是呢，会返回一个ChannelFuture实例来告知你请求的IO操作是成功，失败，还是取消。

channles 是分层的

根据Channel的创建方式，它可以是有parent的(不知道怎么翻)。比如，一个被ServerSocketChannel接收的SocketChannel，会通过parent()来返回一个ServerSocketChannel。

The semantics of the hierarchical structure depends on the transport implementation where the Channel belongs to. For example, you could write a new Channel implementation that creates the sub-channels that share one socket connection, as BEEP and SSH do.

Downcast to access transport-specific operations

Some transports exposes additional operations that is specific to the transport. Down-cast the Channel to sub-type to invoke such operations. For example, with the old I/O datagram transport, multicast join / leave operations are provided by DatagramChannel.

Release resources

It is important to call close() or close(ChannelPromise) to release all resources once you are done with the Channel. This ensures all resources are released in a proper way, i.e. filehandles.

大致流程

当Netty连接到一个客户端或者绑定一个服务器的时候，他需要知道该怎么处理接收或者要发送的数据，这个功能就要被各种各样的Handlers来处理。Netty使用ChannelInitializer来处理这些handlers,他的作用是使用ChannelPipeline来添加ChannelHandler. 当你发送或者接收信息的时候，handlers会决定怎么来处理相关信息。ChannelInitializer自己同样是一个ChannelHandler,当他完成使命在ChannelPipeline中添加了其他ChannelHandler之后，就会把自己从ChannelPipeline中移除。

所有的Netty应用都是基于ChannelPipeline的，ChannelPipeline和EventLoop和EventLoopgroup常常一起出现，因为他们都是和event或者event handling相关的。
Eventloop在应用中的作用是为Channel处理IO操作的，一个EventLoop一般来说会为多个channel来处理event,EventLoopGroup能够包含多个EventLoop,被用作得到EventLoop. Channel代表着一个socket连接或者一些能执行IO操作的组件。因此管理着他的EventLoop的工作是处理IO.

Netty中所有的操作都是异步的，比如说你连接上了一个主机，这个操作默认就是异步的，当你读写数据时候同样也是异步的。这就代表着操作并不会立即返回，而是会延时返回。因为异步，你就无法得知当返回的时候操作是否成功了。所以你需要注册一个监听器，这样当操作完成的时候你就会被通知到了。
为了实现这个功能，Netty使用了Future和ChannelFuture.这个future能够注册监听，当操作结束的时候会通知你，无论操作是否成功。

ChannelHandler

一般情况下我们不直接使用Channel,而是通过ChannelHandler来操作Channel.

来看看API:

public interface ChannelHandler
Handles or intercepts a ChannelInboundInvoker or ChannelOutboundInvoker operation, and forwards it to the next handler in a ChannelPipeline.
Sub-types

ChannelHandler itself does not provide many methods. To handle a a ChannelInboundInvoker or ChannelOutboundInvoker operation you need to implement its sub-interfaces. There are many different sub-interfaces which handles inbound and outbound operations. But the most useful for developers may be:

ChannelInboundHandlerAdapter handles and intercepts inbound operations
ChannelOutboundHandlerAdapter handles and intercepts outbound operations
You will also find more detailed explanation from the documentation of each sub-interface on how an event is interpreted when it goes upstream and downstream respectively.
The context object

A ChannelHandler is provided with a ChannelHandlerContext object. A ChannelHandler is supposed to interact with the ChannelPipeline it belongs to via a context object. Using the context object, the ChannelHandler can pass events upstream or downstream, modify the pipeline dynamically, or store the information (using AttributeKeys) which is specific to the handler.

State management

A ChannelHandler often needs to store some stateful information. The simplest and recommended approach is to use member variables:
public interface Message {
// your methods here
}

public class DataServerHandler extends SimpleChannelInboundHandler {

private boolean loggedIn;

@Override
public void channelRead0(ChannelHandlerContext ctx, Message message) {
    Channel ch = e.getChannel();
    if (message instanceof LoginMessage) {
        authenticate((LoginMessage) message);
        loggedIn = true;
    } else (message instanceof GetDataMessage) {
        if (loggedIn) {
            ch.write(fetchSecret((GetDataMessage) message));
        } else {
            fail();
        }
    }
}
...

}

Because the handler instance has a state variable which is dedicated to one connection, you have to create a new handler instance for each new channel to avoid a race condition where a unauthenticated client can get the confidential information:
// Create a new handler instance per channel.
// See ChannelInitializer.initChannel(Channel).
public class DataServerInitializer extends ChannelInitializer {
@Override
public void initChannel(Channel channel) {
channel.pipeline().addLast(“handler”, new DataServerHandler());
}
}

Using AttributeKey

Although it’s recommended to use member variables to store the state of a handler, for some reason you might not want to create many handler instances. In such a case, you can use AttributeKeys which is provided by ChannelHandlerContext:
public interface Message {
// your methods here
}

@Sharable
public class DataServerHandler extends SimpleChannelInboundHandler {
private final AttributeKey auth =
AttributeKey.valueOf(“auth”);

@Override
public void channelRead(ChannelHandlerContext ctx, Message message) {
    Attribute<Boolean> attr = ctx.attr(auth);
    Channel ch = ctx.channel();
    if (message instanceof LoginMessage) {
        authenticate((LoginMessage) o);
        attr.set(true);
    } else (message instanceof GetDataMessage) {
        if (Boolean.TRUE.equals(attr.get())) {
            ch.write(fetchSecret((GetDataMessage) o));
        } else {
            fail();
        }
    }
}
...

}

Now that the state of the handler isattached to the ChannelHandlerContext, you can add the same handler instance to different pipelines:
public class DataServerInitializer extends ChannelInitializer {

private static final DataServerHandler SHARED = new DataServerHandler();

@Override
public void initChannel(Channel channel) {
    channel.pipeline().addLast("handler", SHARED);
}

}

The @Sharable annotation

In the example above which used an AttributeKey, you might have noticed the @Sharable annotation.

If a ChannelHandler is annotated with the @Sharable annotation, it means you can create an instance of the handler just once and add it to one or more ChannelPipelines multiple times without a race condition.

If this annotation is not specified, you have to create a new handler instance every time you add it to a pipeline because it has unshared state such as member variables.

This annotation is provided for documentation purpose, just like the JCIP annotations.

种类

可以看一下Channel的子接口，分为两种：ChannelInboundHandler和ChannelOutBoundHandler.

ChannelPipeline

基本介绍

ChannelPipeline实际上应该叫做ChannelHandlerPipeline. 他是多个ChannelHandler的集合，负责处理或者拦截到达的事件，Channel发出的操作。ChannelPipeline实现了一个高等的拦截器，这个拦截器会给用户以全面的控制，包括一个事件是怎么被处理的，和其他的ChannelHandler是怎样互动的。 每一个Channel都有一个和他对应的ChannelPipeline. 通过channel.pipeline()获得pipeline，反之使用pipline.channel（）获得Channel.

生成：

每一个Channel里都有ChannelPipeline,他在Channel生成的时候自动生成。要用直接调用就可以了。

ChannelPipeline的事件流

channelEvent分为两种，upstream event和downstream event. 如果在管道中从第一个处理器游向最后一个处理器，我们管它叫上游时间（upstream），反之叫下游事件。

具体说，服务器接收了一个事件，那么叫上游事件，发送了一个事件，叫做下游事件。
同理适用于client，接收事件为上游事件，发送事件为下游事件。

一个inbound event是被图标左侧从下到上的inbound handler处理的。Inbound handeler一般用来处理图表下方的I/O thread生成的inboud data。Inbound data一般是从一个远程端读入的数据。远程端的属于往往是一个真实的写入数据的操作，比如说：SocketChannel.read(ByteBuffer).如果一个inbound event超出了最上方的一个inbound handler，那么他就会被丢弃。你也可以加入日志来统计他们的状态。

同理，一个outbound event是被outbound handler来处理的。对应的是图表中右侧从上到下的flow.一个outbound handler通常生成或者转化outbound traffic,比如说是’写’请求。如果一个outbound event越过了最下面的outbound handler, 他就会被I/O thread和Channel一起处理。 I/O therad一般会执行一个输出动作，比如说SocketChannel.erite(ByteBuffer)

ChannelFuture

刚才看例子，里面server.bind(port).sync()的返回值是ChannelFuture.那么这个ChannelFut是什么呢？

异步通道IO操作的结果。 Netty的所有IO操作都是异步的，这代表着任何的IO请求都会立即返回结果，无论请求是否正确返回。这个返回的结果就是ChannelFuture的实例，这个实例里包含IO操作的结果信息。

ChannelFturn要么是完成的，要么是未完成的。当一个IO请求开始的时候，一个全新的future对象就被创造出来的。新的future在初始化的时候是未完成的，并不是成功，失败或取消中的任何一种状态。如果IO操作结束了，无论他是成功，失败，或者取消，future的状态都是完成的，而且附加了很多用户需要的信息，比如说为什么失败。请注意的是，无论是失败还是取消，他们都属于完成的状态。

Netty中的几个概念

Bootstrap or ServerBootstrap

  • EventLoop: 和channel一起处理event
  • EventLoopGroup:EventLopp线程池
  • ChannelPipeline:ChannelHandler的集合
  • Channel：套接字的通道
  • Future or ChannelFuture：Netty是异步的，用户请求返回的结果
  • ChannelInitializer：ChannelHandler的初始化接口，添加ChannelHandler,添加之后把自己从pipeline中移除。
  • ChannelHandler and its sub-types :处理event的各种类。
