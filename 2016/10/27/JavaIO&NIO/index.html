Java IO & NIO

一次面试关于NIO的东西被问蒙了， 现在来补补课吧！

Linux下的5种线程模型

因为我们的服务器一般情况下都是linux的，我们在这来探讨下线程模型，以理解概念为主。

文件描述符

在Linux下一切皆文件，文件描述符是内核为文件所创建的索引，所有的IO操作都通过调用文件描述符（索引）来执行，包括下面我们要提到的socket.Linux刚启动的时候会自动设置0是标准输入，1是标准输出，2是标准错误。

程序空间与内核空间

在Linux中，对于一次读IO的操作，数据并不会直接拷贝到程序的程序缓冲区，首先他会被拷贝到内核的缓冲区，然后会从内核的缓冲区拷贝到程序的缓冲区.
这个概念很重要，我们来说一遍：

  1. Waiting for the data to be ready (等待数据到达内核缓冲区)
  2. Copying the data from the kernel to the process(从内核缓冲区拷贝数据到程序缓冲区)

阻塞与非阻塞

很容易理解，阻塞就是说我们某一个请求不能立即得到返回值，否则就可以理解为非阻塞。

5种网络IO模型

  1. 阻塞IO：一直等待直到数据返回之前，线程都是block的。
  2. 非阻塞IO：当我们设置一个socket为非阻塞，相当于告诉内核当我们请求的IO操作不能立即返回得到结果，不要把进程设置为sleep状态，而是返回一个错误信息
  3. IO复用(IO multiplexing)：IO复用的好处是我们可以通过（select/epoll/poll）一个时刻处理多个文件描述符。我们可以看到这里有两个return，可以和blocking IO作对比。
  4. 信号驱动IO
  5. 异步IO

select/pool/epool

在Linux上进程/线程之间的切换依然还是很大一笔开销，这个时候我们可以考虑使用上面提到的多路IO复用，他的原理是：一个进程可以监视多个文件描述符,一旦某个文件描述符就绪（读/写就绪）,能够通知程序进行相应的读写操作。

select

对应上面的方法所示，select监听三类描述符:readset,writeset，exceptionset。我们变成的时候可以制定这三个参数监听对应的文件描述符。正如前面提到的，select调用后进程会堵塞，当select返回后，可以通过遍历fd_set来找到就绪的描述符。

select的优点在于它的跨平台，但是显著缺点是单个线程可以监视的文件的描述符存在最大限制值，默认1024/2048. 虽然可以设置超过这个限制，但是效率会变差，而且select扫描的时候需要采用轮询，算法复杂度为O(n）。

下面总结下select的三个缺点，之后我们看看epool是怎么解决这些缺点的：

  1. 每次调用select，都需要将fd_set从用户态拷贝到内核态。
  2. 每次调用select都要在内核遍历传递过来的所有fd_set看哪些状态已经准备就绪。
  3. 有1024的容量限制。

pool

poll和select并没有太大的区别，但是它是基于链表实现的所以并没有最大数量限制，它将用户传入的数据拷贝到内核空间，然后查询每个fd对应的设备状态，如果设备就绪则在设备等待队列中加入一项并继续遍历，如果遍历完所有fd后没有发现就绪设备，则挂起当前进程，直到设备就绪或者主动超时，被唤醒后它又要再次遍历fd。这个过程经历了多次的遍历。算法复杂度也是O(n)。

epoll

select和poll都只提供了一个函数，而epoll提供了三个：epoll_create是创建一个epoll句柄,epoll_ctl是是注册要监听的事件类型，epoll_wait则是等待时间的产生。 与select相比，epoll几乎没有描述符的限制大小(cat /proc/sys/fs/file-max可查看)。它采用一个文件描述符管理多个描述符，将用户的文件描述符事先存放到kernel的一个事件列表中。这样在程序空间和内核空间只需要做一次拷贝，他去掉了遍历文件符这一步骤，采用更加先进的回调机制，算法复杂度降低到了O(1).
ps:虽然表面看起来epoll非常好，但是对于少数连接而且十分活跃的情况下，select和poll的性能可能比epoll好，因为epoll是建立在大量的回调函数之上。

总结一下epoll是如何解决select的三个缺点的：

  1. 直接将文件符存储到内核里，只需要一次拷贝
  2. epoll_ctl为每个fd指定一个回调函数，当设备就绪就会调用这个回调函数，而这个回调函数会把准备就绪的fd加入一个就绪链表，而不用像select那样去重新遍历一次看有哪些准备就绪的文件描述符。
  3. 无大小限制 (cat /proc/sys/fs/file-max)

NIO

IO一看也没什么太细节的东西，都是读了就了解了，咱们说说NIO.

Java NIO有三个核心部分

  • Selector
  • Channel
  • Buffer

Channel and Buffer

C和B之间和可传递数据，他们都有不同的实现

Channel

  • FileChannel
  • DataGramChannel
  • SocketChannel
  • ServerSocketChannel

Buffer
就是一些数据类型的buffer

  • ByteBuffer
  • ShortBuffer
  • DoubleBuffer
  • LongBuffer
  • IntBuffer
  • CharBuffer
  • FloatBuffer

Selector
Selector可以在单线程下管理多个channel，要使用selector,你需要向Selector注册Channel,然后调用他的seek()，这个方法会一直阻塞到他注册的通道有事件就绪，然后返回，这样线程就会处理这个事件。

more about Channel

既可以从Channel读数据，又可以写数据到Channel.Channel可以进行异步的读写。Channel中的数据总是要写入一个buffer中，或者从一个buffer中读，就像上面图示的那样。

我们刚说了有四种channel

FileChanner: 主要负责从文件中读取数据,是阻塞IO。
DataGramChannel: 通过UDP读写数据
SocketChannel: 通过TCP读写数据
ServerSocketChannel: 监听新的TCP连接，每有一个连接就注册一个SocketChannel

刚看buffer的代码有一点蒙，缓了那么一下，才想起来他的工作机制，buffer有几个属性，我们需要了解一下：

  • capcity: buffer的容量大小，一般用buffer的子类的allocate(num)方法来定义大小。
  • limit: buffer可用的大小。
  • position: 下一个需要读取元素的索引。
  • mark: 一个记录当前position的值，当使用mark()的时候会设置成mark=position,调用reset方法可以使position返回到mark的位置。
